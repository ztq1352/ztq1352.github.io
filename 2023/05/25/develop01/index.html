后端开发学习总结(一)

发表于2023-05-25|更新于2023-05-25|学习总结
|字数总计:2.9k|阅读时长:12分钟|阅读量:

mybatis开发流程

  1. 根据需求分析功能并确认各功能的开发顺序

    例如：用户注册功能由用户数据新增，以及用户数据查询组成，功能开发顺序：查询用户是否存在–>存在则插入用户数据

  2. 规划需要执行的sql语句

    用户注册前需要根据用户名查询数据是否已经存在
    select * from t_user where username
    用户注册，插入新增用户信息
    insert into t_user ( username,password,salt…) values(“张三”,”123”…);

  3. 设计接口和抽象方法

    定义mapper接口，在项目的目录结构创建mapper包，在这个包下根据不同的功能模块创建mapper接口，创建UserMapper接口。

  4. 配置映射

    在resources目录下创建一个mapper文件夹存放xxxMapper.xml映射文件
    映射文件名称与mapper接口名称保持一致。
    配置文件配置映射文件路径：mybatis.mapper-locations=classpath:mapper/*Mapper.xml

  5. 测试

    在测试包下创建mapper包，包下创建xxxMapperTests类
    添加@SpringBootTest和@RunWith(SpringRunner.class)注解
    注入xxxMapper，使用@Test注解测试方法

业务层开发流程

  1. 分析业务层异常并处理异常

    分析功能实现会出现的异常，创建ex包并创建业务层异常基类继承RuntimeException，具体的异常类继承基类，并生成异常类中定义的构造方法
    例如：在用户注册功能实现中可能出现数据插入过程所产生的异常、用户名被占用异常

  2. 设计业务层接口和抽象方法,并分析需求实现需求

    根据业务需求创建IxxxService接口类与xxxServiceImpl具体实现类。
    分析业务需求，根据业务需求分步骤逐步实现需求

  3. 单元测试

    在测试包下创建service包，包下创建xxxServiceTests类
    添加@SpringBootTest和@RunWith(SpringRunner.class)注解
    注入xxxService，使用@Test注解测试方法

控制层开发流程

  1. 分析控制层异常并处理异常

    考虑业务层抛出的异常是什么？
    需要统一异常处理类中进行统一的捕获和处理，如果已创建统一异常处理类。则不需要继续添加，直接创建具体异常类继承

  2. 设计请求

    请求路径 /users/login
    请求方式 POST
    请求数据 String username, String passoword, HttpSession session
    响应结果 JsonResult

  3. 处理请求

    在控制类中编写处理请求的方法

  4. Postman测试

    GET请求：数据在Params下添加，会自动填充url末尾
    POST请求：数据在Body下的根据提交类型添加
    需要携带的session数据在Cookies添加
    测试:localhost:8080/users/reg?username=lisi&password=123
    {
    “state”: 200,
    “message”: null,
    “data”: null
    }

用户会话session

session对象主要存在服务器端，可以用于保存服务器的临时数据的对象，所保存的数据可以在整个项目中都可以访问获取，把session的数据看做共享数据。
首次登录的时候所获取的用户数据，转移到session对象中。
session.getAttrbute(“key”)可以将获取session中的数据这种行为进行封装，封装在BaseController类中

  1. 封装session对象的数据获取（封装到父类中），数据的设置（当用户登录成功后进行数据的设置，设置到全部的sesion对象中）。

  2. 在父类中封装两个数据：获取uid和获取username对应的两个方法，头像暂时不考虑，封装在cookie中

拦截器

将所有请求统一拦截器中，拦截器定义过滤规则，如果不满足系统的过滤规则，跳转指定页面（重定向/页面转发）推荐重定向。
使用springmvc提供的HandlerInterceptor接口，用于定于一个拦截器，受限制的自定义类，实现这个接口。

  1. 自定义类，实现HandlerInterceptor接口

  2. 注册过滤器：添加白名单（不拦截页面）、黑名单

    借助WebMvcConfigure接口，可以将用户定义的拦截器进行注册，才能保证拦截器能够生效和使用。定义一个类实现WebMvcConfigure接口，配置信息，建议存放在config包结构下。
    使用@Configuration注解声明配置注册。
