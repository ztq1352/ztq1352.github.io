MySQL学习总结(一)

发表于2023-05-25|更新于2023-06-07|数据库
|字数总计:5.9k|阅读时长:18分钟|阅读量:

概念

  1. 数据库：DataBase，按照一定格式存储文件的集合，是数据存储的仓库。
  2. 数据库管理系统：DataBaseManagement，用来管理数据库数据的系统，对数据库数据进行操作。
  3. SQL语言：是关系型数据库的标准语言，用来存储、检索和修改关系型数据库中存储的数据。
    三者关系：数据库管理系统–执行->SQL–操作->数据库
  4. 常见关系型数据库：MySQL、Oracle、SQL Server等

MYSQL中的数据类型

  • varchar（255内）：可变长度的字符串，根据实际长度动态分配空间，节省空间，效率低
  • char（255内）：定长字符串，不管实际长度，分配固定长度，容易导致空间浪费，效率高
  • int（11内）：数字中的整数型，等于java的int
  • bigint：数字中的长整型，等于java的long
  • float：单精度浮点型
  • double：双精度浮点型
  • date：短日期，年月日，默认格式%Y-%m-%d
  • datetime：长日期，年月日时分秒，默认格式%Y-%m-%d %h:%i:%s
  • clob：字符大对象，最大可以存储4G的字符串，超大255字符使用
  • blob：二进制大对象，专门用来存储图片、声音视频等流媒体数据。要使用IO流

SQL脚本的执行

数据导入

进入mysql命令窗口执行sql脚本：

  1. 进入mysql数据库服务器上
    mysql> mysql -u 用户名 -p (回车后输入密码)
  2. 新建要建的数据库再使用数据库或直接使用已存在的数据库
    mysql> create database 数据库名;
    mysql> use 数据库名
  3. 初始化数据库
    mysql> source D:\xxx\xxx\xxx.sql

数据库导出

  1. 进入MySQL目录下的bin文件夹：cd MySQL中到bin文件夹的目录，或者手动进入bin文件夹使用cmd
    cd C:\xxx\MySQL\MySQL Server 8.0\bin
  2. 输入：mysqldump -u 用户名 -p 数据库名 > 导出的文件名
    mysqldump -u root -p dbtest > dbtest.sql # 输入后输入进入MySQL的密码
    如果导出单张表的话在数据库名后面输入表名即可

SQL语言分类

DDL

数据定义语言：create、drop、alter表结构的增删改（对表结构进行操作）

  1. 查询所有库
    show databases;
  2. 查询指定的库
    show create database 库名
  3. 修改库的字符集
    alter database 库名 default character set 字符集;
  4. 删除库
    drop database 库名;
    drop database if exists 库名;
  5. 新建库
    create database 库名;
    create database if not exists 库名;
  6. 修改表的字段类型
    alter table 表名 modify 当前的字段名称 新的字段类型;
  7. 修改的字段名称
    alter table 表名 change 旧字段名称 新的字段名称 以前的数据类型;
  8. 添加一个新的字段
    alter table 表名 add 字段名称 字段类型;
  9. 删除字段
    alter table 表名 drop 字段名称;
  10. 查询的表的结构
    desc 表名;
  11. 查询库中有哪些表
    show tables;
  12. 复制一张新的表
    create table 新表名 like 旧表名;
  13. 将一个查询结果当作一张表新建
    create table 新表名 as select * from 要复制的表名

DML

数据操作语言：insert、delete、update表数据的增删改（对表当中的数据进行增删改）

  • insert语法结构如下：
    insert into 表名 (字段1,字段2...) values(值1,值2...),(第二条记录)....; #字段跟值一一对应
    注意：
    insert语句但凡执行成功，必然会多一条数据
    insert语句字段名可以省略，但是值必须都要写上

  • delete语法结构如下：
    delete from 表名 where 条件;

  • update语法结构如下：
    update 表名 set 字段名1=值1，字段名2=值2... where 条件;

  • 将查询结果插入到某一张表中
    insert into 表1 select语句;

DQL

数据查询语言：select表数据的查询（带有select关键字的都是查询语句）
所有的sql语句以分号结尾，且sql语句不区分大小写。
在所有数据库当中，字符串以单引号括起来，是标准的。
select后面可以跟某个字段名（可以等同看做变量名），也可以跟字面量/字面值（数据）

1. 单表查询

  • 简单查询语句：

      1.单字段查询：select 字段名 from 表名； 
  2.多字段查询使用“,”隔开：select 字段名1,字段名2 from 表名；
  3.查询所有字段使用“*”：select * from 表名；（效率低，可读性差，不建议使用）   
  4.给字段起别名使用“as”关键字：select 字段名 as 别名 form 表名；（只是显示别名，字段名不会被修改）as关键字可以省略，用空格替换（前提是字段名没有空格，字段名有空格需要用单引号或者双引号）   
  5.字段可以使用数字表达式：select 字段名*12 from 表名；

  • 条件查询语句：

      语法：select 字段1,字段2... from 表名 where 条件；
  1.=等于、<>或!=不等于、>大于、<小于、<=小于等于、>=大于等于
  2.between ... and ... 两个值之间（必须遵循左小右大），等同于 >= and <=
  3.is null 为 null（is not null 不为空），在数据库中null不能使用=衡量
  4.and 并且、or或者
      and和or的优先级？	and的优先级大于or，需要使用括号来区分执行先后
  5.in包含（not in），相当于多个or（not or 不在这个范围中），in不是区间，in后面跟着是具体的值
      字段 in(值1,值2)；
  6.not 可以取非，主要用在is或in中
      is null、is not null、 in、not in
  7.like 称为模糊查询，支持%或下划线匹配（%匹配任意个字符，下划线，一个下划线只匹配一个字符）
       like %a%、%t、t%
      like a_、_a
      如果所查数据名含有下划线_,查询含下划线的数据，需要使用\_转义

  • 排序语句：

      order by 字段名；默认升序asc，使用desc设置为降序
  order by 字段1 asc,字段2 asc；第一个为主导，如果存在相等情况下再根据第二个排序

  • 数据处理函数（单行处理函数）：一个输入对应一个输出

      Lower()：转换小写
  upper()：转换大写
  substr(,,)：去字串（substr（被截取的字符串，起始下标(只能从1开始)，截取的长度））
  length()：取长度
  trim()：去空格
  round(,保留位数)：四舍五入
  rand()：生成随机数
  ifnull(,替换值)：可以将null转换成一个具体的值空处理函数，专门处理空的，在所有数据库当中，只要有null参与的数学运算，最终结果就是null
  concat(,)：字符串拼接
  str_to_date(字符串,格式)：将字符串转换成日期，通常在insert语句中使用
  date_format(日期类型字段,格式)：格式化日期，通常在select语句中，
  format(字段，格式$999)：设置千分位
  case..when..then..when..then..else..end

  • 分组函数（多行处理函数）：多个输入，1个输出

      count() 计数
  sum() 求和
  avg() 平均值
  max() 最大值 
  min() 最小值
  注意：分组函数在使用的时候必须先进行分组，然后才能用。如果没有进行分组，默认为整张表
  1.分组函数自动忽略null
  2.分组函数count(*)和count(具体字段)区别？
      count(具体字段)：统计该字段下不为null的总数
      count(*)：统计表总行数（只要有一行数据count则++）
  3.分组函数不能直接使用在where子句中（先分组之后才能使用）
  4.所有的分组函数可以组合起来一起用

  • 分组查询：先分组再查询 

      group by

  • having关键字 

      不能代替where，必须与group by联合使用，where和having优先选择where

多表查询

  • distinct关键字

      把查询结果去除重复记录
  distinct放在所有字段前，表示多个字段联合去重 
  能够跟分组函数进行使用：count（distinct 字段）

  • 连接查询

      多张表联合起来查询数据
  SQL92：1992年出现的语法。结构不清晰，表连接条件跟后期进一步筛选都放在一起
  SQL99：1999年出现的语法

  • 表连接分类

    • 内连接

      分类：等值连接、非等值连接、自连接
内连接特点：完成能够匹配上这个条件的数据查询出来
内连接之等值连接(连接条件相等)：表1 inner(可省略，带着可读性好) join 表2 on 连接条件
内连接之非等值连接(连接条件不是等值关系)
内连接之自连接：将一张表看做两张表，表连接表

    • 外连接

      分类：左外连接(左连接)、右外连接(右连接)
right /left 表示将join关键字左/右边的这张表看成主表，主要是为了将这张表的数据全部查询出来，捎带着关联查询右/左边的表
右外连接：right outer(可省略，带着可读性好) join
左外连接：left outer(可省略，带着可读性好)  join
外连接查询的数据条数一定是>=内连接的查询的数据条数

    • 全连接

  • 笛卡尔积现象

      两张表进行连接查询，没有任何限制，数据条数会出现两张表数据条数的的乘积，连接时加条件，满足这个条件的记录被筛选出来。匹配次数没少，
  通过笛卡尔积现象得出，表的连接次数越多效率就越低，尽量避免表的连接次数

  • 子查询

      select语句嵌套select语句
  1.select后嵌套（基本不用），子查询只能接收返回的一条数据，多一条就报错
  2.from后嵌套 from后的子查询查出的数据相当于一张临时表，再进行查询
  3.where后嵌套 解决where语句后不能使用分组函数

  • union关键字

      select语句 union select语句
  union效率高一点
  结果集合并要求列相同，数据类型也得相同

  • limit关键字

      将查询结果集的一部分取出来，通常在分页查询中使用
  limit 值：前几的条数
  limit 值1,值2：起始下标0开始，长度
  order by后使用

语法顺序

    关键字顺序：select[distinct] --> from --> join --> on --> where --> group by --> having --> union --> order by -->limit
    执行顺序： from --> on --> join --> where --> group by --> having --> select --> distinct --> union -->order by

TCL

事务控制语言：commit、rollback事务提交回滚

DCL

数据控制语言：授权grant、撤销权限revoke….

约束(constraint)

创建表时给表中的字段添加约束，保证数据的完整性、有效性。
非空约束：not null，字段不能为空 ，只有列级约束，没有表级约束
唯一性约束：unique，字段不能重复，但是可以为空
主键约束：primary key，即不能为空且是唯一性，称为主键
外键约束：foreign key，
检查约束：check(mysql不支持，oracle支持)
约束添加在字段后为 列级约束
约束添加在所有字段后，为表级约束

  1. 两个字段联合具有唯一性？ 

     两个字段后添加unique是错误的，他们是各自唯一性。
 字段定义完后添加，正确unique(字段1，字段2)

  2. 非空约束和唯一性约束可以联合使用

     mysql自动变成主键约束，在oracle不是

  3. 主键约束术语

     主键约束：约束
 主键字段：添加了主键约束的字段
 主键值：主键字段下的每一个值
 注意：主键值是每一行记录的唯一标识！任何一张表都必须有主键
 特征：not null+unique
 表级约束：primary key(字段名1，字段名2)
 单一主键、复合主键（实际开发中不建议使用）
 一张表主键约束只能添加1个
 主键值建议使用：int、bigint、char等
 不建议varchar，一般主键用数字，而且是定长

  4. 自然主键

     自然数，和业务没关系

  5. 业务主键

     与业务紧密相关，不建议使用，业务主键值受业务影响

  6. 主键自增

     primary key auto_increment：主键自增，从1开始

  7. 外键约束相关术语

     外键约束：一种约束
 外键字段：字段上添加了外键约束
 外键值：外键字段的每一个值
 为了保证表1的字段所添加的数据跟表2的字段值有关，不会出现数据无效，因此添加外键约束
 表1为子表，表2为父表
 如果删除数据，需要先删除子表再删除父表，创建父表再创建子表，先插入父表再插入子表

  8. 外键引用

     表级位置--> foreign key(父表字段名) references 表2(表2字段名)  
 子表中的外键字段引用的父表的字段必须是主键吗？
      错误，引用父表的字段至少具有唯一性约束
 子表中的外键值可以为null

存储引擎

    存储引擎是MySQL中特有的一个术语，其它数据库中没有。（Oracle中有，但是不叫这个名字）
    实际上存储引擎是一个表存储/组织数据的方式。
    不同的存储引擎，表存储数据的方式不同

    CREATE TABLE `xxx` (...)ENGINE=InnoDB DEFAUL CHARSET=utf8
    在建表的时候可以在最后小括号的")"的右边使用：
    ENGINE来指定存储引擎。 mysql默认的存储引擎是：InnoDB
    CHARSET来指定这张表的字符编码方式。mysql默认的字符编码方式是：utf8

mysql支持的存储引擎

    show engines \G 查看mysql支持哪些存储引擎

MyISAM存储引擎

  • 使用三个文件表示每个表：
    • 格式文件 — 存储表结构的定义（mytable.frm）
    • 数据文件 — 存储表行的内容（mytable.MYD）
    • 索引文件 — 存储表上索引（mytable.MYI）：索引是一本书的目录，缩小扫描范围，提高查询效率的一种机制。
  • 可被转换为压缩、只读表来节省空间
  • MyISAM存储引擎特点：
    • 可被转换为压缩、只读表来节省空间这是这种存储引擎的优势！！！！
    • MyISAM不支持事务机制，安全性低。

InnoDB存储引擎

  • 这是mysql默认的存储引擎，同时也是一个重量级的存储引擎。
  • InnoDB支持事务，支持数据库崩溃后自动恢复机制。
  • InnoDB存储引擎最主要的特点是：非常安全。
  • 它管理的表具有下列主要特征：
    • 每个 InnoDB 表在数据库目录中以.frm 格式文件表示
    • InnoDB 表空间 tablespace 被用于存储表的内容（表空间是一个逻辑名称。表空间存储数据+索引。）
    • 提供一组用来记录事务性活动的日志文件
    • 用 COMMIT(提交)、SAVEPOINT 及ROLLBACK(回滚)支持事务处理
    • 提供全 ACID 兼容
    • 在 MySQL 服务器崩溃后提供自动恢复
    • 多版本（MVCC）和行级锁定
    • 支持外键及引用的完整性，包括级联删除和更新
  • InnoDB最大的特点就是支持事务：以保证数据的安全。效率不是很高，并且也不能压缩，不能转换为只读，
  • 不能很好的节省存储空间。

MEMORY存储引擎

  • 使用 MEMORY 存储引擎的表，其数据存储在内存中，且行的长度固定，这两个特点使得 MEMORY 存储引擎非常快。
  • MEMORY 存储引擎管理的表具有下列特征：
    • 在数据库目录内，每个表均以.frm 格式的文件表示。
    • 表数据及索引被存储在内存中。（目的就是快，查询快！）
    • 表级锁机制。
    • 不能包含 TEXT 或 BLOB 字段。
  • MEMORY 存储引擎以前被称为HEAP 引擎。
  • MEMORY引擎优点：查询效率是最高的。不需要和硬盘交互。
  • MEMORY引擎缺点：不安全，关机之后数据消失。因为数据和索引都是在内存当中。

事务

  1. 什么是事务？ 

     事务就是一个完整的业务逻辑，是一个最小的工作单元，不可再分。(案例：转账-->同时成功或者同时失败)

  2. 只有DML语句跟事务有关，其他语句和事务无关！

     insert、delete、update（增删改）-->需要考虑安全问题

  3. 为什么需要存在事务机制？

     因为某个业务不能通过一条DML语句完成，需要多条DML语句协同联合才能完成，需要同时成功或者同时失败。

  4. 事务是怎么做到多条DML语句同时成功和同时失败呢？

     开启事务后有两种方法结束事务：提交事务（commit）和回滚事务（rollback）
 提交事务：清空事务性活动日志文件，将执行了所有DML语句的操作数据彻底全部持久化。（同时成功）
 回滚事务：将执行了的DML语句的全部撤销，并且清空事务性活动日志文件。（同时失败）

  5. 事务的默认机制是什么？

     mysql默认情况下是事务是自动提交的，没执行一次DML语句就提交一次。
 关闭事务使用start transaction语句关闭自动提交机制。

  6. 事务的特性有哪些？

     原子性：说明事务是最小的工作单元，不可再分
 一致性：同一个事务中， 所有的操作必须同时成功或者同时失败
 隔离性：两个事务之间具有一定的隔离性
 持久性：事务结束的保障，事务提交保障数据持久化到硬盘。

  7. 隔离性：隔离级别有哪些？

     读未提交：read unconmmitted 事务A可以读取到事务B未提交的数据。（出现脏读现象）
 读已提交（oracle默认级别）：read commit 事务A只能读取到事务B提交之后的数据。（解决脏读现象，出现不可重复读数据现象）
 不可重复读：事务A开启事务后读取事务B的数据第一次跟读取第二次的数据不同。
 可重复读（mysql默认级别）：respeatable read 事务A开启事务后，不管读取多少次事务B的数据都是相同的，不管事务B数据是否发生改变。（出现幻读现象：每次读取的数据不真实）
 序列化、串行化（效率最低）：serializable 事务排队，不能并发。

索引

  1. 什么是索引？

     索引是在表的字段上添加，是为了提高查询效率存在的一种机制		
 一张表的一个字段可以添加一个索引，多个字段也可以联合起来添加索引
 索引相当于一本书的目录，是为了缩小扫描范围而存在的一种机制
 创建索引：create index 索引名 on 表名(字段名)
 删除索引：drop index 索引名 on 表名

  2. 查找数据的方式？

     一页一页挨着找：全扫描
 通过目录(索引)定位大概位置，定位到位置后开始局域性扫描：索引检索，效率较高

  3. 索引(index)扫描

     在mysql当中，索引是需要排序的，使用B-Tree数据结构
 遵循左小右大原则存放，采用中序遍历方式遍历取数据

  4. 索引原理

     缩小扫描范围，避免全表扫描

  5. 索引失效的情况

     1.模糊查询匹配以“%”开始，需要尽量避免
 2.使用or的时候，两边字段都需要有索引才能成功，其中一边没有索引则失效
 3.使用复合索引的时候，没有使用左侧的列查找，索引失效
 4.在where当中索引列参加了运算，索引失效
 5.在where当中，索引列使用了函数，索引失效

  6. 索引分类

     单一索引
 复合索引
 主键索引
 唯一性索引
 注意：唯一性比较弱的字段上添加索引用处不大

  7. 添加索引的条件

     1.数据量庞大
 2.该字段经常出现在where后面，以条件的形式存在，也就是说该字段经常被扫描
 3，该字段很少DML操作(因为DML之后，索引需要重新排序)
 建议不要随意添加索引，索引需要维护
 建议通过主键或者unique约束的字段查询

  8. 提醒

     1.在任何数据库当中，主键都会自动添加索引对象
 2.在任何数据库当中，任何一张表的任何一条记录在硬盘存储上都有一个硬盘的物理存储编号
 3.在mysql当中，索引是一个单独的对象，不同的存储引擎以不同的形式存在，
   在myisam中，存储在.myi文件中，
   在innodb中，存储在一个逻辑名称tablespace中，
   在memory中，存储在内存中，
 无论存储在哪里，mysql都以树的形式存在，自平衡二叉树(B-tree)
 在mysql当中，主键上，以及unique字段上都会自动添加索引

视图

view：站在不同的角度，看待同一份数据
视图存储在硬盘中
创建视图：create view 视图名称 as select语句;
删除视图：drop view 视图名称;
注意：只有DQL语句才能以view形式创建

视图能做什么

面向视图对象进行增删改查，且视图对象的增删改查会导致原表被操作！
面向视图增删改查都会影响

视图在实际开发中作用

简化：复杂的，需要反复使用的sql语句，利于后期维护

数据库三范式

数据库表的设计依据
第一范式：要求任何一张表必须有主键，每一个字段原子性不可再分
第二范式：在第一范式基础上，要求所有非主键字段完全依赖主键，不要产生部分依赖
第三范式：在第二范式基础上，要求所有非主键字段直接依赖主键，不要产生传递依赖
注意：按照三范式可以避免数据库冗余，空间浪费。

口诀

多对多，三张表，关系表两外键
一对多，两张表，多的表加外键
一对一，外键唯一

2. truncate与drop,delete的对比

  • 回滚
    • truncate与drop是DDL语句，执行后无法回滚；
    • delete是DML语句，可回滚。
  • 作用点
    • truncate只能作用于表；
    • delete，drop可作用于表、视图等。
  • 删除效果
    • truncate会清空表中的所有行，但表结构及其约束、索引等保持不变；
    • drop会删除表的结构及其所依赖的约束、索引等。
  • 自增
    • truncate会重置表的自增值；delete不会。
  • 关于触发器
    • truncate不会激活与表有关的删除触发器；delete可以。
  • 空间释放：
    • truncate后会使表和索引所占用的空间会恢复到初始大小；
    • delete操作不会减少表或索引所占用的空间；
    • drop语句将表所占用的空间全释放掉。
